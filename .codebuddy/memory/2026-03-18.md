# 2026-03-18

## 修复：创建新节点失败

### 根因分析
1. **ID 生成器排序 bug（核心原因）**：`id-generator.ts` 中 `generateTaskId` 使用 `orderBy(desc(tasks.id))` 按自增主键排序来查找最大编号，但如果有任务被删除后重新创建，主键最大的记录不一定有最大的编号后缀，导致生成重复 taskId，触发 UNIQUE 约束失败。
2. **并发竞态条件**：快速连续创建两个节点时，两次调用都读到相同的最大编号，生成相同的 taskId。
3. **缺少错误处理**：`POST /api/v1/tasks` 路由没有 try/catch，异常直接抛出 500。

### 修复内容
- `server/src/services/id-generator.ts`：改为遍历所有匹配行取最大数字编号 + 加互斥锁防并发
- `server/src/api/routes.ts`：为创建节点 API 添加 try/catch 错误处理
- `server/src/services/task-service.ts`：domain/milestone 查询添加 projectId 条件（修复多项目同名问题）

## 修复：创建节点仍然失败（真正根因）

### 根因
`schema.ts` 中定义了 `assignee`（处理人）字段，但 `connection.ts` 的 `runMigrations()` 中既没有在 `CREATE TABLE tasks` 中包含该列，也没有 `ALTER TABLE tasks ADD COLUMN assignee` 迁移语句。导致实际 SQLite 数据库中不存在 `assignee` 列，Drizzle ORM 生成的 INSERT 语句引用该列时报错：`table tasks has no column named assignee`。

### 修复
- `server/src/db/connection.ts`：
  1. 在初始 `CREATE TABLE tasks` 中加入 `assignee TEXT` 列（新建库生效）
  2. 在增量迁移区域追加 `ALTER TABLE tasks ADD COLUMN assignee TEXT` 和 `ALTER TABLE tasks ADD COLUMN start_date TEXT`（旧库升级生效）

## 修复：创建根节点"不成功"（实际上是创建成功但 UI 未刷新）

### 根因
各个页面使用的 React Query queryKey 不一致：
- TaskList: `['task-tree-list', ...]`
- KanbanBoard: `['task-tree-kanban', ...]`
- MindMap/Requirements: `['task-tree', ...]`
- MyTasks: `['my-task-tree', ...]`

而 `CreateTaskModal` 的 `onSuccess` 回调只 invalidate 了 `['tasks']` 和 `['task-tree']`，不会匹配 `task-tree-kanban`、`task-tree-list`、`my-task-tree` 等其他变体。
结果：创建成功、弹窗关闭，但页面数据未刷新——用户以为创建失败。

### 修复
- `CreateTaskModal.tsx`：改用 `predicate` 匹配所有以 `task` 或 `my-task` 开头的 queryKey 来 invalidate
- `TaskDetail.tsx`：同样修复 `invalidate()` 和子节点创建后的刷新
- `BatchActionBar.tsx`：同样修复批量操作后的刷新
- 同时为 `CreateTaskModal` 添加了 `onError` 回调和错误提示 UI

## 修复：UNIQUE constraint failed: tasks.task_id（ID 生成器按 projectId 过滤导致全局冲突）

### 根因
`id-generator.ts` 的 `_doGenerateTaskId()` 在查找最大编号时用 `projectId` 过滤：
```
const conditions = [like(tasks.taskId, `${prefix}-%`)];
if (projectId) conditions.push(eq(tasks.projectId, projectId));
```
但 `tasks.task_id` 的 UNIQUE 约束是**全局跨项目的**。如果多个 project 使用同一前缀（如 `T`），只在单个 project 范围内查最大编号会导致跨项目 ID 冲突。

### 修复
- `id-generator.ts`：移除 `projectId` 过滤，改为**全局查找**该前缀下的最大编号
- 添加**防御性冲突检查**：生成 ID 后再查一次确认不存在，如果冲突则自动递增重试
- 同样修复了 `generateBacklogId`

### 第一轮结果：117/151 通过，34 失败

### 发现并修复的 Bug

1. **Fastify `FST_ERR_CTP_EMPTY_JSON_BODY`（34 个测试失败的主因）**
   - 根因：测试脚本所有请求都带 `Content-Type: application/json`，但 DELETE/POST 等无 body 的请求会被 Fastify 拒绝（空 JSON body 400 错误）
   - 修复：测试脚本的 `api()` 函数在无 body 时不设置 `Content-Type` 头

2. **Reparent 循环检测失败（`_isDescendant` 参数反了）**
   - 根因：`task-service.ts` 的 `reparent()` 方法调用 `_isDescendant(parent.id, task.id)` 检查循环，但 `_isDescendant` 实际上是从 startId 的子树中查找 targetId。参数顺序应该是 `(task.id, parent.id)` 来检查 new parent 是否在当前节点的子树中
   - 修复：交换参数为 `_isDescendant(task.id, parent.id)`，同时修正了函数注释和参数命名

3. **deleteTask 缺少 objectiveTaskLinks 清理（外键约束导致 500）**
   - 根因：`deleteTask()` 方法清理了 taskNotes/progressHistory/taskFieldValues/taskAttachments/taskIterations，但没有清理 `objective_task_links` 表（该表没有 `ON DELETE CASCADE`）。当被删除的 task 有 objective link 时，外键约束失败导致 500
   - 修复：在 `deleteTask()` 中增加清理 `objectiveTaskLinks`、`reqLinks`、`taskPermissions` 的逻辑

### 最终结果：153/153 全部通过（连续两轮验证稳定）
