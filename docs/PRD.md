## 一、背景与愿景

### 1.1 核心痛点

| 痛点 | 描述 |
|------|------|
| **进度黑箱** | 多个 Agent 各自开发，PM 无法实时掌握各模块进展 |
| **需求散落** | 决策者口头提的需求、Agent 发现的需求，没有统一收纳 |
| **创建摩擦大** | 传统 PM 工具要求填一堆字段才能创建需求，打断思维流 |
| **层级僵化** | Epic/Story/Task/Subtask 固定四层结构，实际使用中产生认知摩擦 |
| **Agent 不友好** | 缺少面向 AI Agent 的 MCP 接口，Agent 无法自主领取和上报 |
| **无可视化** | 没有思维导图、看板、甘特图等直观的项目全景视图 |

### 1.2 产品愿景

> **ClawPM = 思维导图式的需求树 + Human-in-the-Loop 的 GUI + Agent 友好的 MCP 接口**

核心思想：**需求管理应该像思维导图软件一样畅通无阻**。一个快捷键就创建好一个节点，先上车后补票——节点创建后可以慢慢完善细节，不需要在创建时填一堆繁文缛节。

```
┌─────────────────────────────────────────────────────────┐
│                 ClawPM — 项目管理中枢                      │
│                                                         │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐        │
│  │ 人类决策者 │────►│ 思维导图   │◄────│ AI Agent  │        │
│  │  Web GUI  │     │ 需求树核心 │     │  MCP 接口 │        │
│  └──────────┘     └──────────┘     └──────────┘        │
│                                                         │
│  底层逻辑：思维导图式需求树 = 项目管理的数据中枢             │
│  上层交互：GUI（人看/管） + MCP（Agent 读/写）             │
└─────────────────────────────────────────────────────────┘
```

### 1.3 v2.0 设计哲学

1. **一切皆节点** — 没有 Epic/Story/Task/Subtask 之分，所有需求节点本质平等，只是粒度不同
2. **先上车后补票** — 创建节点只需要一个标题，甚至只需按一个 Tab 键；未完善的节点自动进入"未排期"状态
3. **思维导图为核心** — 主视图是思维导图，节点的创建、编辑、组织都以思维导图交互为中心
4. **标签而非类型** — 用可选的自定义标签（如 #epic #bug #feature）进行分类，不作为系统级约束
5. **AI 可拆解** — 支持 AI 根据一个高层需求自动拆解生成子节点树
6. **双通道交互** — GUI 给人用（Human-in-the-Loop），MCP 给 Agent 用（自动化流水线）

## 三、核心概念

### 3.0 项目（Project）— v2.1 新增

**项目是 ClawPM 中最顶层的数据隔离单元。** 所有业务数据（节点、板块、里程碑、需求池、目标、成员）都归属于某个项目，项目之间数据完全隔离。

```
Project（项目）
├── slug         URL 友好的短标识（唯一，如 "letsgo"、"clawpm"）
├── name         显示名称（如 "LetsGo 项目"）
├── description  简介（可选）
├── archived     是否归档（默认 false）
├── created_at   创建时间
└── updated_at   更新时间

数据隔离范围：
├── tasks          所有节点归属项目
├── domains        业务板块按项目隔离
├── milestones     里程碑按项目隔离
├── backlog_items  需求池按项目隔离
├── goals          OKR 目标按项目隔离
├── members        成员按项目隔离（同一人可属于多个项目）
└── custom_fields  自定义字段按项目隔离
```

#### 3.0.1 项目管理功能

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| PJ-01 | 项目列表 | 展示所有项目卡片，显示名称、节点数、最近活跃时间 | **P0** |
| PJ-02 | 创建项目 | 填写 name 和 slug（slug 自动生成可编辑） | **P0** |
| PJ-03 | 项目切换 | 侧边栏顶部或全局导航栏的项目选择器，一键切换当前项目 | **P0** |
| PJ-04 | 项目设置 | 编辑名称、描述、归档 | P1 |
| PJ-05 | 归档项目 | 标记归档后从列表隐藏，可恢复 | P2 |

#### 3.0.2 交互设计

1. **首次使用**：如果没有任何项目，自动创建一个名为"默认项目"（slug: `default`）的项目
2. **项目选择页**：访问根路径 `/` 显示项目列表，选择后进入项目工作区
3. **项目切换器**：侧边栏顶部固定显示当前项目名称，点击弹出项目列表快速切换
4. **URL 结构**：所有页面路径包含项目 slug，如 `/p/letsgo/board`、`/p/letsgo/mindmap`
5. **MCP 交互**：所有 MCP 工具增加可选的 `project` 参数（传 slug），不传时使用默认项目

#### 3.0.3 数据隔离原则

- 切换项目后，所有视图（看板、思维导图、任务列表、仪表盘等）只展示当前项目的数据
- 搜索、筛选、统计均限定在当前项目范围内
- 成员可以属于多个项目（未来可扩展跨项目视图）

### 3.1 需求节点（Node）

系统中所有的需求、任务、想法统一为"节点"（Node）。节点之间通过父子关系构成树状结构，任何节点都可以有无限个子节点，没有深度限制。

```
需求节点
├── title        标题（唯一必填字段）
├── status       状态（见 3.2）
├── labels       可选标签数组（如 ["epic", "用户系统"]）
├── priority     优先级 P0/P1/P2/P3（默认 P2）
├── progress     进度 0–100（默认 0）
├── owner        负责人（可选）
├── description  详细描述（可选）
├── due_date     截止日期（可选）
├── parent_id    父节点 ID（null = 根节点）
├── [children]   子节点列表（递归）
└── [attachments] 附件列表：文档/链接/TAPD（v2.2 新增）
```

### 3.2 节点状态体系（核心）

v2.0 重新定义了需求的状态流转，更贴合实际工作流：

```
快速创建（Tab/Enter）
       │
       ▼
┌──────────────┐
│   未排期      │  刚创建的节点，只有标题，尚未规划
│  (backlog)   │  → 想法池：先记下来，后续完善
└──────┬───────┘
       │ 完善细节并排入迭代
       ▼
┌──────────────┐
│   未开始      │  已排期，分配了负责人，等待启动
│  (planned)   │
└──────┬───────┘
       │ Agent 或人开始执行
       ▼
┌──────────────┐
│   进行中      │  正在开发/执行
│  (active)    │
└──────┬───────┘
       │ 完成开发，提交验收
       ▼
┌──────────────┐
│   验收中      │  产品验收 / 代码评审
│  (review)    │
└──────┬───────┘
       │ 验收通过
       ▼
┌──────────────┐
│   已完成      │  归档，更新统计
│  (done)      │
└──────────────┘
```

**关键设计决策：**
- 新创建的节点默认状态为 `backlog`（未排期），而不是 `planned`
- 去掉了 `blocked`（阻塞）和 `cancelled`（已取消）作为核心状态——这两个通过标签或 `blocker` 字段表达
- 5 个状态精简明确，映射清晰：想法 → 排期 → 执行 → 验收 → 完成

**看板映射：**

| 状态 | 看板列名 | 颜色 | 说明 |
|------|---------|------|------|
| `backlog` | 未排期 | 灰色 | 想法池，待完善 |
| `planned` | 未开始 | 蓝色 | 已排期，等待执行 |
| `active` | 进行中 | 靛蓝 | 正在执行 |
| `review` | 验收中 | 琥珀 | 待验收 |
| `done` | 已完成 | 绿色 | 已归档 |

### 3.3 标签系统（Labels）

标签是节点的**可选**分类标注，不影响节点的行为。系统预置推荐标签，用户可自由扩展：

| 推荐标签 | 说明 | 使用场景 |
|----------|------|---------|
| `epic` | 大功能方向 | 顶层需求聚合 |
| `feature` | 用户可见特性 | 一个完整的功能模块 |
| `bug` | 缺陷修复 | 问题追踪 |
| `spike` | 技术预研 | 调研、原型验证 |
| `chore` | 工程维护 | 重构、升级、部署 |

> **核心原则：** 标签仅用于分类和过滤，不限制节点的行为。一个打了 `epic` 标签的节点可以没有子节点，一个没有标签的节点也可以有 100 个子节点。

### 3.4 关联（Association）

除了父子树状关系，节点之间还支持横向关联：

| 关联类型 | 方向 | 语义 | 思维导图显示 |
|----------|------|------|------------|
| **阻塞依赖** (`blocks`) | 单向 | B 未完成，A 无法开始 | 红色虚线 + 箭头 |
| **顺序依赖** (`precedes`) | 单向 | 推荐先做 B 再做 A | 橙色虚线 + 箭头 |
| **弱关联** (`relates`) | 双向 | 主题相关，无依赖 | 蓝色虚线 |

关联规则：
1. 阻塞/顺序依赖不允许形成环路（系统自动检测）
2. 关联线可按类型独立显示/隐藏
3. 移动节点（换父节点）时关联保持不变

### 3.5 节点附件系统（v2.2 新增）

每个节点都支持挂载**附件**，用于记录相关文档、外部链接和第三方工单关联。附件让节点从一个"标题+状态"的轻量卡片，变成一个**信息中枢**——需求文档、设计文档、TAPD 单、Wiki 链接都能直接从节点快速查看和跳转。

#### 3.5.1 附件类型

| 类型 | 标识 | 说明 | 示例 |
|------|------|------|------|
| **Markdown 文档** | `doc` | 节点内嵌的富文本文档，支持完整 Markdown 语法 | 需求文档、设计文档、技术方案、会议纪要 |
| **外部链接** | `link` | 指向外部 URL 的快捷方式，可一键跳转 | Wiki 页面、Figma 设计稿、飞书文档、Git 仓库 |
| **TAPD 关联** | `tapd` | 关联 TAPD 的需求/缺陷/任务，自动展示 TAPD 单状态 | TAPD 需求 #12345、TAPD 缺陷 #67890 |

#### 3.5.2 数据模型

```
TaskAttachment（节点附件）
├── task_id       归属节点
├── type          类型: doc | link | tapd
├── title         显示名称（如 "需求文档"、"设计稿链接"）
├── content       内容（Markdown 文本 / URL / TAPD ID）
├── metadata      扩展元数据 JSON（如 TAPD 类型、URL 图标等）
├── sort_order    排序序号（同一节点内排序）
├── created_by    创建人
├── created_at    创建时间
└── updated_at    更新时间
```

#### 3.5.3 功能需求

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| AT-01 | 添加 Markdown 文档 | 节点详情页可添加多个 MD 文档，支持实时编辑预览 | **P0** |
| AT-02 | 添加外部链接 | 输入 URL + 标题，显示为可点击的链接卡片 | **P0** |
| AT-03 | 添加 TAPD 关联 | 输入 TAPD 单 ID 或 URL，自动解析并展示 TAPD 单摘要 | **P1** |
| AT-04 | 附件列表 | 节点详情侧边栏展示所有附件，按类型分组 | **P0** |
| AT-05 | 快速预览 | 点击 MD 文档附件展开预览，无需跳转 | **P0** |
| AT-06 | 一键跳转 | 点击链接/TAPD 附件直接在新标签页打开 | **P0** |
| AT-07 | 排序拖拽 | 拖拽调整附件顺序 | P2 |
| AT-08 | 附件搜索 | 支持在附件内容中搜索关键词 | P2 |

#### 3.5.4 交互设计

1. **节点详情页**：附件区域位于节点描述（description）下方，分三个 Tab：「文档」「链接」「TAPD」
2. **Markdown 文档**：
   - 点击文档标题展开/收起预览区（渲染后的 Markdown）
   - 点击"编辑"进入编辑器模式（左右分栏：编辑 + 实时预览）
   - 支持多个文档，如一个节点同时挂"需求文档"和"技术方案"
3. **外部链接**：
   - 显示为卡片：图标 + 标题 + URL 域名 + 点击跳转
   - 链接类型自动识别：iWiki、飞书、Figma、GitHub 等显示对应图标
4. **TAPD 关联**：
   - 输入 TAPD 单 ID 或完整 URL，自动解析
   - 显示 TAPD 单摘要：标题、状态、负责人（通过 TAPD MCP 工具获取）
   - 点击直接跳转到 TAPD 页面
5. **思维导图集成**：节点卡片上显示附件计数图标（如 📎3），表示该节点有 3 个附件
6. **MCP 集成**：Agent 可通过 MCP 工具为节点添加/查询附件

#### 3.5.5 典型使用场景

```
节点「用户登录模块」
├── 📄 需求文档.md       ← 完整的需求描述、验收标准
├── 📄 技术方案.md       ← 接口设计、数据流程
├── 🔗 Figma 设计稿      ← https://figma.com/file/xxx
├── 🔗 iWiki 会议纪要    ← https://iwiki.woa.com/pages/xxx
├── 🎫 TAPD 需求 #12345  ← 关联的 TAPD 需求单
└── 🎫 TAPD 缺陷 #67890  ← 关联的 TAPD 缺陷单
```

> **设计理念**：节点是信息中枢，所有相关文档和链接都应该能从节点一键触达，而不是散落在各处。

## 四、思维导图视图（核心交互）

### 4.1 设计原则

思维导图是 ClawPM 的**主视图**。它的 UX 目标是"像 XMind / MindNode 一样丝滑"：

1. **自动布局** — 节点位置由布局算法自动计算，用户不需要手动摆放
2. **子随父动** — 移动根节点时，整个子树跟随移动
3. **快捷键驱动** — Tab 创建子节点、Enter 创建同级、Delete 删除、双击改名
4. **内联创建** — Tab/Enter 后直接在思维导图上出现一个空白节点，输入标题后 Enter 确认即完成创建，无需弹窗
5. **折叠展开** — 子树可折叠/展开，折叠时显示子节点计数
6. **右键菜单** — 添加关联、折叠/展开、删除等高级操作

### 4.2 交互流程

```
思维导图操作流程

选中节点
  ├── Tab      → 内联创建子节点（出现空白节点，输入标题，Enter 确认）
  ├── Enter    → 内联创建同级节点
  ├── Delete   → 删除确认
  ├── 双击      → 内联编辑标题
  ├── 右键      → 上下文菜单（添加关联/折叠/删除/更多…）
  └── + 按钮   → 同 Tab，添加子节点

新节点创建后：
  → 状态自动设为 backlog（未排期）
  → 进入思维导图，可以慢慢完善详情（点击节点打开侧边栏编辑）
  → 不完善也不影响任何功能，先记下来就好
```

### 4.3 思维导图功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 内联创建节点 | Tab/Enter 即时创建，无弹窗阻塞 | **P0** |
| 自动树形布局 | 横向树形布局，子节点自适应位置 | **P0** |
| 子树跟随移动 | 拖拽根节点时整棵子树跟随 | **P0** |
| 折叠/展开 | 折叠子树，显示子节点计数气泡 | **P0** |
| 双击编辑标题 | 内联编辑，Enter 确认 Esc 取消 | **P0** |
| 状态色点 | 节点上显示状态对应的彩色圆点 | **P0** |
| **核心字段筛选** | 右侧面板支持按状态/优先级/负责人/里程碑/标签/时间范围筛选节点 | **P0** |
| 关联线显示 | 阻塞/顺序/弱关联线，可按类型开关 | P1 |
| 右键菜单 | 添加子节点/同级/关联/删除等操作 | P1 |
| 新建根节点 | 按钮创建新的根节点（新的树） | P1 |
| AI 自动拆解 | 选中节点 → 调用 AI 自动生成子节点树 | P2 |

### 4.4 思维导图筛选面板

节点作为基础类，所有可设置参数都应在思维导图中提供筛选能力。筛选面板位于右侧控制面板，提供以下核心筛选维度：

| 筛选维度 | 类型 | UI 组件 | 说明 |
|----------|------|---------|------|
| **状态** (`status`) | 多选 | 复选按钮组 | backlog/planned/active/review/done，默认全选 |
| **优先级** (`priority`) | 多选 | 复选按钮组 | P0/P1/P2/P3 |
| **负责人** (`owner`) | 单选 | 下拉选择 | 从 members 列表中选择，含"未指定"选项 |
| **里程碑** (`milestone`) | 单选 | 下拉选择 | 从 milestones 列表中选择 |
| **标签** (`labels`) | 单选 | 按钮组 | epic/feature/bug/spike/chore + 自定义标签 |
| **截止日期** | 范围 | 起止日期选择器 | 筛选 due_date 在指定范围内的节点 |
| **开始日期** | 范围 | 起止日期选择器 | 筛选 start_date 在指定范围内的节点 |
| **自定义字段** | 动态 | 根据字段类型渲染 | select/multi_select/date/number/text |

**筛选规则：**
1. 多个筛选条件之间是 AND 关系（同时满足所有条件）
2. 同一维度内多选是 OR 关系（如选了 P0+P1，匹配任一即可）
3. 筛选后保留匹配节点的祖先路径（保持树结构完整）
4. 筛选后保留匹配节点的子孙路径（如果父节点匹配，子节点全部显示）
5. 提供"清除全部"按钮一键重置所有筛选条件
6. 筛选状态持久化到 localStorage

### 4.4 AI 自动拆解需求（v2.0 规划）

选中一个节点后，通过 MCP 调用 AI Agent 自动拆解：

```
用户选中节点「用户系统」
  │
  ├── 右键 → "AI 拆解"
  │
  ▼
AI Agent 通过 MCP 调用：
  create_task("注册流程", parent="用户系统")
  create_task("登录流程", parent="用户系统")
  create_task("第三方 OAuth", parent="登录流程")
  create_task("权限管理", parent="用户系统")
  ...
  │
  ▼
思维导图自动刷新，显示 AI 生成的子树
```

> 拆解由 Agent 通过 MCP 完成，ClawPM 本身不包含 AI 能力。

### 6.3 description 字段约定

`description` 字段用于记录成员的**擅长领域、职责范围和工作偏好**，建议内容包括：

- **擅长领域**：如"前端开发、React"、"后端架构、数据库优化"
- **负责板块**：如"用户系统"、"支付模块"
- **工作偏好**：如"擅长快速原型"、"适合深度重构"
- **特殊说明**：如 Agent 的能力边界

> **设计原则**：description 是自由文本，不做结构化约束。AI Agent 通过 MCP 查询成员列表时可获取此字段，结合 `taskCount`（总任务数）和 `activeCount`（活跃任务数）来做智能任务分配。

### 6.5 典型使用场景


## 十三、协作与个人工作台（v2.3 新增）

### 13.1 设计理念

> **灵魂**：以需求树为底层驱动逻辑的需求管理系统，所有需求都是一个叶子节点，需求应该是清晰知道来源的，整个项目就是一棵树，基于树的根节点衍生开枝散叶。

个人工作台不是扁平的任务列表，而是**"我的需求子树"**——从全局需求树中提取所有分配给我的节点及其祖先路径，让每个成员清晰看到：

1. **任务从哪来** — 它在树上的位置、它的父节点（上游需求）是什么
2. **任务为了什么** — 沿着树往上回溯，最终指向哪个根节点目标
3. **我在整棵树中的位置** — 我做的这片叶子和全局的关系

### 13.2 F1: 轻量级身份识别

**核心思想**：不做密码/登录系统，只做"身份声明"——轻量、零摩擦。

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| ID-01 | 身份选择 | 首次访问弹出身份选择面板，从已有成员列表选择"我是谁" | **P0** |
| ID-02 | 身份持久化 | 选择后将 identifier 持久化到 localStorage，后续请求自动附带 | **P0** |
| ID-03 | 身份切换 | 侧边栏底部显示当前身份，支持点击切换 | **P0** |
| ID-04 | 快速创建身份 | 身份选择面板支持快速创建新成员 | P1 |

**交互设计**：
- 首次访问时全屏遮罩弹出选择面板，列出当前项目成员（头像+名称+类型标签）
- 点选即确认，零摩擦
- 侧边栏底部显示当前身份头像+名称，点击可切换

### 13.3 F2: 个人工作台——"我的需求子树"

**核心设计**：从全局需求树中，提取所有 `owner === 我` 的节点及其完整祖先路径，以缩进树形结构展示。

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| MY-01 | 我的需求子树 | 调用需求树 API（owner 过滤），以缩进树形展示，我的节点高亮、祖先节点半透明 | **P0** |
| MY-02 | 面包屑溯源 | 每个"我的节点"显示从根节点到当前节点的完整路径 | **P0** |
| MY-03 | 顶部统计 | 进行中、待验收、未开始、已逾期的任务数统计 | **P0** |
| MY-04 | 快速推进状态 | hover 时显示"推进"按钮，一键推到下一个合法状态 | P1 |
| MY-05 | 侧边栏入口 | "工作台"分组下新增"我的任务"导航项 | **P0** |

**展示规则**：
- 我负责的节点（`owner === currentUser`）：正常高亮显示，左侧靛蓝竖条标记
- 祖先上下文节点：半透明显示（opacity-50），不可操作，仅作为路径上下文
- 空状态：友好提示"暂无分配给你的任务"

### 13.4 F3: MCP Agent 身份自动绑定

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| AG-01 | Agent 身份绑定 | MCP stdio 模式支持通过 `CLAWPM_AGENT_ID` 环境变量或 `--agent-id` 参数传入身份 | **P0** |
| AG-02 | get_my_tasks 自动填充 | `owner` 参数变为可选，未传时 fallback 到 session 身份 | **P0** |
| AG-03 | whoami 工具 | 新增 MCP 工具，Agent 可查询自己的身份信息 | **P0** |
| AG-04 | get_my_task_tree | 新增 MCP 工具，返回带祖先路径的个人需求子树 | P1 |

### 13.5 F4: 仪表盘增强——个人视角

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|

## 十四、节点权限控制 (v2.4)

### 14.1 设计理念

每个任务节点都有唯一的 **Owner**（负责人），Owner 是节点的完全控制者。Owner 可以将节点的访问权限授予其他成员，权限分为两个等级：

| 权限等级 | 标识 | 能力范围 |
|----------|------|----------|
| **可编辑** (`edit`) | 橙色编辑图标 | 在该节点下增/删/改子节点、修改节点状态、修改节点字段 |
| **可查看** (`view`) | 灰色眼睛图标 | 仅能查看节点内容（标题、描述、状态、附件等），不可修改任何信息 |

### 14.2 权限规则

| 规则 | 描述 |
|------|------|
| **Owner 拥有全部权限** | Owner 无需额外授权即拥有该节点及其所有子节点的完全控制权 |
| **权限不自动继承** | 父节点的权限不自动传导到子节点；每个节点独立管理权限 |
| **Owner 不可转让（仅可重新指派）** | Owner 身份通过修改节点的 `owner` 字段变更，不通过权限系统转让 |
| **被授权人须为项目成员** | 只能将权限授予已存在的 `members` 中的成员（identifier） |
| **同一被授权人不可重复授权** | 对同一节点，一个被授权人只能持有一条权限记录；重复授权覆盖旧等级 |
| **Owner 可随时撤销权限** | Owner 可以撤销任何人对该节点的授权 |
| **无权限 = 不可见** | 对于开启了权限控制的节点，非 Owner 且无授权的成员在"我的需求子树"中不会看到该节点内容（但祖先路径仍可展示为脱敏占位） |

### 14.3 权限控制的开启方式

节点默认**不开启权限控制**（即公开可见，保持现有行为）。当 Owner 首次为某节点添加权限授权时，该节点自动进入"受控"状态。当所有授权被撤销时，节点恢复"公开"状态。

> **设计原则**：不给现有使用流程增加任何额外操作。权限是可选的增强能力，不使用时零影响。

### 14.4 功能列表

| ID | 功能 | 描述 | 优先级 |
|----|------|------|--------|
| PM-01 | 授予权限 | Owner 在任务详情中可为其他成员授予 `edit` 或 `view` 权限 | **P0** |
| PM-02 | 撤销权限 | Owner 可撤销已授予的权限 | **P0** |
| PM-03 | 查看权限列表 | 任务详情中显示当前节点的所有被授权人及其权限等级 | **P0** |
| PM-04 | 权限等级变更 | Owner 可将已授权人的权限从 `view` 升级为 `edit` 或从 `edit` 降级为 `view` | **P0** |
| PM-05 | 写操作拦截 | 非 Owner 且无 `edit` 权限的用户尝试修改节点时，返回 403 错误并提示"无编辑权限" | **P0** |
| PM-06 | 前端只读适配 | `view` 权限用户看到的任务详情中，所有编辑按钮/输入框置灰或隐藏 | P1 |
| PM-07 | 权限标识展示 | 在"我的需求子树"和任务详情中，展示当前用户对该节点的权限等级标识 | P1 |
| PM-08 | MCP 权限工具 | 新增 `grant_permission` / `revoke_permission` / `list_permissions` MCP 工具 | P1 |

### 14.5 交互设计

**任务详情 — 权限管理面板**：
- 仅 Owner 可见"权限管理"入口（锁图标）
- 展开后显示已授权人列表：头像 + 名称 + 权限等级下拉 + 删除按钮
- 底部"添加协作者"：从成员列表中选择 + 选择权限等级
- 权限变更即时生效

**非 Owner 用户视角**：
- `edit` 权限：正常显示编辑功能，但不可管理权限、不可删除节点
- `view` 权限：所有可编辑区域变为只读展示，隐藏"推进状态"等操作按钮
- 无权限：节点不在"我的需求子树"中显示，或展示为"[受限节点]"占位

## 十七、Intake 收件箱 (v3.1)

### 17.1 设计理念

> **Intake = 项目的"收件箱"**，让外部人员（测试同学、用户、其他团队成员）提交 Bug 报告、功能建议或一般反馈，项目成员审核后决定接受/拒绝/暂缓，接受后自动转为正式需求节点。

参考 Plane 的 Intake（原 Inbox）机制，但做了以下 ClawPM 化适配：
1. **与标签系统联动** — 提交时选择类别（Bug/Feature/Feedback），接受后自动打上对应标签
2. **与需求树融合** — 接受后直接变为需求树节点，可指定父节点，继承 ClawPM 的"一切皆节点"理念
3. **无需登录即可提交** — 提交者只需填写名字，无需注册账号（降低提交门槛）
4. **审核人可丰富信息** — 接受时可同时设置优先级、负责人、父节点等

### 17.2 Intake 条目模型

```
IntakeItem（收件箱条目）
├── intake_id    业务 ID（如 "IN-001"）
├── project_id   归属项目
├── title        标题（必填）
├── description  详细描述（支持 Markdown）
├── category     类别: bug | feature | feedback
├── submitter    提交人名称（自由填写，非 member）
├── status       审核状态: pending | accepted | rejected | deferred | duplicate
├── priority     建议优先级（提交者填写，审核者可修改）
├── reviewed_by  审核人（member identifier）
├── review_note  审核备注（拒绝/暂缓时的理由）
├── task_id      转化后的节点 ID（accepted 时关联）
├── created_at   提交时间
└── updated_at   更新时间
```

### 17.5 交互设计

#### 17.5.1 提交表单（公开页面）

- **URL**：`/p/:slug/intake/submit`（无需认证，公开可访问）
- **表单字段**：
  - 标题（必填）
  - 类别选择：🐛 Bug 报告 / ✨ 功能建议 / 💬 一般反馈
  - 详细描述（Markdown 编辑器，可选）
  - 提交人名称（必填，自由填写）
  - 建议优先级（P0/P1/P2/P3，默认 P2，可选）
- **提交成功**：显示确认页面，包含 Intake ID 供后续查询
- **设计风格**：简洁、友好、无干扰，引导提交者清晰描述问题

#### 17.5.2 Intake 管理列表（项目成员）

- **URL**：`/p/:slug/intake`
- **默认视图**：按状态分组展示，pending 置顶高亮
- **卡片信息**：标题、类别标签、提交人、提交时间、建议优先级
- **筛选**：按状态、类别筛选
- **待审核 badge**：侧边栏"收件箱"导航项显示 pending 数量

#### 17.5.3 审核操作

点击 Intake 条目展开详情，审核面板提供：

| 操作 | 描述 | 后续动作 |
|------|------|---------|
| ✅ 接受 | 认可该提交，转为正式节点 | 弹出接受配置面板：选择父节点、设置负责人、调整优先级 → 创建 Task |
| ❌ 拒绝 | 不予采纳 | 必填拒绝理由 |
| ⏸️ 暂缓 | 有价值但现在不做 | 选填暂缓理由 |
| 🔄 重复 | 与已有节点重复 | 可关联到已有 Task ID |

#### 17.5.4 接受配置面板

接受时弹出配置面板，允许审核者丰富信息：
- **父节点**：下拉选择，默认为根节点
- **负责人**：从成员列表选择（可选）
- **优先级**：可调整提交者的建议优先级
- **额外标签**：除了类别自动标签外，可追加更多标签
- 确认后自动创建节点，Intake 条目关联到新节点

### 17.6 Intake 与现有系统的关系

| 维度 | 关系 |
|------|------|
| **与需求池的区别** | 需求池（Backlog）是内部成员录入的需求想法；Intake 是外部/开放提交的入口，需要审核才能进入系统 |
| **与标签系统的联动** | 接受时根据 category 自动打标签：bug→`bug`，feature→`feature`，feedback→`feedback` |
| **与通知系统的联动** | 新 Intake 提交时通知项目成员；审核完成可通知相关人 |
| **与 MCP 的集成** | AI Agent 可通过 MCP 提交 Intake（如从飞书消息自动提取 Bug 报告） |
